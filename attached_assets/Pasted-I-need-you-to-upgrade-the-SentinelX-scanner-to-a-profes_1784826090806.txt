## 1. FIX SSTI FALSE POSITIVES (CRITICAL)

The current SSTI check in `vuln-probes.ts` → `checkSSTI()` only looks for `49` anywhere in the response body. That produces false positives on dynamic pages (e.g., Cloudflare Ray IDs). Replace that logic with:

a) **Reflection gate**: Before even searching for the result, confirm the *raw payload* (e.g., `{{7*7}}` or `{{'SENTINELX_SSTI_' + 'CONFIRM'}}`) actually appears in the response body. If it’s not reflected, do NOT flag SSTI for that parameter.

b) **Proximity matching**: If the payload is reflected, find its position(s) in the body. Then look for the expected result (`49`) within a window of 200 characters around each reflection. Only count matches near the reflection.

c) **Unique canary**: As the primary verification, send `{{'SENTINELX_SSTI_' + 'CONFIRM'}}` (URL‑encoded). If the exact string `SENTINELX_SSTI_CONFIRM` appears in the response body AND was absent from the baseline, mark the finding as **VERIFIED** with high confidence. If this canary fails, downgrade the finding or suppress it entirely.

d) **Multi‑expression confirmation**: After the first math probe, immediately send a second request with `{{8*8}}`. If the response contains `64` (and `49` disappears or reduces), increase confidence. Only report SSTI if *both* math probes pass *and* the payload is reflected, **or** the canary succeeds.

e) **Dynamic‑content filter**: Before any string matching, strip out known ephemeral tokens from the response body:
   - Cloudflare Ray IDs (regex: `[a-f0-9]{16,}-[A-Z]{3}`)
   - `__cf_bm` cookie values
   - Challenge nonces, timestamps like `2026-07-23T16:34:12.865Z`
