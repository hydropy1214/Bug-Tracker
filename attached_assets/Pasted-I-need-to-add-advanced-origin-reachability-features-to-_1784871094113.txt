I need to add advanced origin‑reachability features to my authorized security testing tool, SentinelX. Many of our clients use CDN/WAF services like Cloudflare. During authorized assessments, it’s critical that our scanner can test the origin server directly, either because the client provides the origin IP or because we can discover it through passive means. This is standard practice in tools like Burp Suite and Nuclei.

Please implement the following capabilities. They should only activate when the scan profile is set to "deep_authorized" (the mode we use for fully consented penetration tests).

## 1. Manual Origin Override
Allow the user to optionally provide an origin IP or hostname in the scan configuration. When set, all scan phases (port scanning, TLS analysis, path discovery, injection tests, etc.) will route requests to that IP using the original domain’s Host header. This lets us test the real server when the CDN interferes with our authorized tests.

## 2. Passive Origin Discovery (crt.sh & DNS History)
When no manual origin is given, attempt to discover the origin IP using only passive sources:
- Query crt.sh (Certificate Transparency) for the domain and extract any IPv4 addresses that appear in certificate CN/SAN fields.
- Use DNS resolvers to check if the domain ever resolved to an IP before it was behind the CDN (e.g., by looking at historical A records from public DNS databases, if an API key is configured – if not, skip this step).
- Collect all candidate IPs and deduplicate them.

## 3. Semi‑Passive Confirmation
For each candidate IP (max 10), send a single HTTP GET request with the correct Host header and a timeout of 3 seconds. Check if the response:
- Lacks CDN‑specific headers like cf-ray, cf-cache-status, x-amz-cf-id, etc.
- Differs significantly from the CDN challenge page.
If both conditions are met, mark that IP as a confirmed origin and automatically use it for all subsequent scan phases (just like the manual override).

## 4. Request‑Variation Tests (Only on Confirmed Origin)
When a confirmed origin is available, add a small set of harmless request variations to ensure our tests can reach the application:
- Try a few legitimate User‑Agent strings (Googlebot, Chrome, etc.) to avoid bot‑blocking.
- Add common IP‑spoofing headers (X-Forwarded-For, X-Real-IP, CF-Connecting-IP, True-Client-IP) with internal IPs like 127.0.0.1 or 10.0.0.1. This is a standard test to see if the origin trusts these headers, as misconfigurations are common.
- If none of these change the response, just continue with the normal requests; the origin is still reachable.

## 5. Integration with Scanning Phases
When an origin IP is in use (manual or discovered), all subsequent phases should direct their traffic to that IP with the correct Host header. The WAF‑detection gate should be considered bypassed, allowing the safe vulnerability verification engine (Phase 24) to run normally against the origin.

## Important Safeguards
- These features must only operate when the scan profile is "deep_authorized". Default safe scans will not perform origin discovery or use overrides.
- Only a maximum of 10 candidate IPs will be tested, with a single lightweight request each, to avoid any disruption.
- All actions are logged clearly so the tester can see exactly what happened.
- No actual exploitation occurs during discovery; it simply finds a direct line to the server that the tester is already authorized to test.

This is a standard feature in professional penetration testing platforms, and it will make SentinelX much more effective for authorized assessments where CDNs are involved. Please implement it step by step, keeping all existing safety checks intact.
